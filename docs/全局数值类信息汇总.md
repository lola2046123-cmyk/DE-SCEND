## 0. v0.06 经济改造（已落地阶段）

### 0.0 生涯档案 · 博彩对账三件套（基础合规层）

为支撑 v0.06 经济模型的"赢/亏对账"，`fbc_career_archive_v1` 在 0.05 五字段之上扩 **2 项**：

| 字段名（JSON） | 含义（玩家侧） | 写入 / 更新公式 |
|----------------|----------------|----------------|
| **`total_buy_in`**（v0.06 新增） | 累计建仓投入：每局玩家在授权面板点"启动电梯"确认的杠杆之和 | `+= floor(amount)`；调用点：`index.html` · `btnDeploy.click` 内 `GameController.careerMerge({ buyIn: amount })`。 |
| **`total_credits_lost`**（v0.06 新增） | 累计破产损失：清算事件中累计该局 `initialBet` | `+= initialBet`；调用点：`finishReveal()` 未缓和 LIQUIDATION 与 `_autoLiquidate()`（Hiss 超时）共两处，与 `liquidation: true` 同 delta 写入。 |
| **`total_quota_crossings`**（v0.06 升级 B 新增） | 累计配额过线次数（每局至多 +1） | `+= 1`；调用点：`finishReveal()` 内 `_quotaCrossed` 翻位为 true 的同一帧，与 `quotaCrossed: true` 同 delta 写入。 |
| **`total_debt_cashouts`**（v0.06 升级 C-1 新增） | 累计未达配额触发的债务撤离次数 | `+= 1`；调用点：`forceDebtCashOut()` 内与 `creditsCollected: payout` 同 delta 写入。 |

**衍生量**（不入库，UI 实时计算）：

```
net P&L = total_credits_collected − total_buy_in
```

**右上角 ≡ 战绩页 渲染顺序**（`index.html` · `renderCareer`）：

| 行 | 内容 | 视觉 |
|----|------|------|
| 1 | `累计建仓投入` / `累计核销资产` 并列 | 暗金 / FBC 暖色 |
| 2 | `综合损益结算 · NET P&L`（通栏） | 正绿 `#94d0a8` / 负红 `#db8778` / 平 `#c8d0d8` |
| 3 | `历史最高深度` / `累计上行楼层` 并列 | — |
| 4 | `破产记录: N 次 · 损失 ¥X`（通栏） | `>=3 次` 时红色 |
| 5 | `FBC 委员会代决议次数`（通栏） | — |

**兼容性**：`careerLoad` 对缺失键回退 0；旧档案（无 `total_buy_in` / `total_credits_lost`）打开战绩页时 PnL 显示为 `+¥<已核销资产>`，下一次 deploy 起恢复对账。

### 0.1 baselineGrowth 分段衰减（House Edge 改造 ①）

`config/game-events.json` · `baselineGrowth.tiers` 定义按 **stakeRatio = credits / quota** 的衰减曲线，调用点为 `js/game-controller.js` · `_applyBaselineGrowth` → `pickBaselineGrowthTier(credits, quota)`。当 `tiers` 缺省或全为非法值时回退至 `multiplier`。

| stakeRatio < | multiplier | 区间含义 |
|---|---|---|
| 1.5 | **1.10** | 未达配额 1.5×，全力激励 |
| 3.0 | **1.06** | 已盈利 1×：增长被压低 |
| 6.0 | **1.03** | 暴富区：每层只涨 3% |
| ≥ 6.0 | **1.015** | 富翁陷阱：守住比赚到更难 |

> 实例可读：当前 tier 标签存于 `this._lastBaselineTier`（label ∈ `incentive | throttle | freeze | millionaire-trap | flat`），用于 HUD/日志诊断。

### 0.2 FBC 流通损耗（House Edge 改造 ②）

`config/game-events.json` · `fbcEdge` 在 `_applyOutcomeToCredits` 出口对 `delta > 0` 的正向收益分级抽水；亏损与清算不抽税。税额计入 `outcome._fbcEdgeAmount` 与 `outcome._fbcEdgeRate`，`index.html` · `showDoorResultFrame` 在三位一体帧底部追加一行极细官僚注脚：

```
// FBC §4.7 已扣 ¥XX · YY% 流通损耗
```

| 结果带 | 默认税率 | RTP 影响 |
|---|---|---|
| POSITIVE | **0.04** | 单层 RTP −2.4% |
| DOUBLE | **0.08** | 单层 RTP −1.1% |
| GOLDEN（POSITIVE 且 `goldenFloor`） | **0.12** | 肥尾大赢另税 |
| NEGATIVE / LIQUIDATION | 0 | 输了不再加税 |

抽水时机在 `speculator` / `blindConfidence` 等正向乘子之后执行，语义为"FBC 对所有渠道的正面收益统一切片"。`fbcEdge.enabled=false` 时整段 no-op。

### 0.3 动态配额（升级 A）

`config/game-events.json` · `quota.tiers` 按 buy-in 分段，将 quota 从静态 ¥1,000 升级为"按建仓规模动态浮动"。读取入口 `js/game-controller.js` · `pickQuotaForBuyIn(buyIn)` 在每次 `reset()` 用 `this.initialBet` 求值。

| `buyInMax` | `multiplier` | tier label | 含义 |
|---|---|---|---|
| 250 | **5.0** | small-stake | 小注：5× 难门槛，逼迫久玩 |
| 600 | **4.0** | standard | 标准：4× 经典 RTP 调试基线 |
| 1500 | **3.0** | leveraged | 高杠杆：3× 风险撞配额 |
| 99999 | **2.5** | high-roller | 大户：相对快达标，给"暴富"留缝 |

- 落地：`this.quota = round(buyIn × multiplier / 50) × 50`，且不低于 `quota.minQuota = 500`。
- 暴露：`GameController.pickQuotaForBuyIn` 给 UI 预览（授权面板的 `// 目标 ¥X · 杠杆 ¥Y` 即用此函数计算）。
- 兼容：`quota.tiers` 为空数组时回退 `quota.target`（旧版默认 1000）。

### 0.4 配额过线庆贺（升级 B · v0.06.2 已精简，v0.06.3 视觉色相对调不变）

`finishReveal()` 在 `_quotaCrossed` 翻位的同一帧 emit `quotaCrossed` 事件，UI 反馈：

1. **HUD 染色**：`#asset-counter` 加 `--quota-cleared`（`#money` / quota / 分隔符渐染 mint）+ `--quota-flash`（一次性 1.1s 扫光）。
2. ~~**mono 旁注**：HUD 数字下方一次性出现 9 px monospace 标签 `// ALLOC_QUOTA · CLEARED`~~ — **v0.06.2 移除**（占用 LED 高度且与第 3 项闪现重复，规则同步删除 `.hud__quota-cleared-tag` / `#hud-quota-cleared-tag` / `quotaClearedTagEl` / `@keyframes fbc-quota-cleared-tag-in`）。
3. **场景闪现**：`stage` 顶部一次性悬浮 `// FBC §3.1 — 配额已达成 ×<multiplier>`，`backdrop-filter: blur(2px)`，2.6s 自销毁。
4. **节拍奖励**：下一次 `LOCKER_SELECTING` 倒计时 `+= quota.crossedBonusMs (1000)`，仅触发一次（`quota.crossedBonusOnce = true`），存于 `this._quotaBonusPendingMs`，由 `_lockerHandMeta.quotaBonusApplied` 报告。
5. **生涯档案**：同帧写 `careerMerge({ quotaCrossing: true })` → `total_quota_crossings += 1`。

清场：`resetVisual()` 调 `clearQuotaClearedFeedback()` 清掉 HUD class 与扫光定时器。

> v0.06.2 视觉补丁同步要点：HUD LED 默认色由暖珊瑚改为 `--hud-led-readout` (`#F2F5F9` 冷白)；mint 仅留给本节"配额过线"等极端态。详见 `docs/设计规范-Design-System.md § 9` 与 `docs/0.06版本更新产品功能.md § 七`。
>
> **v0.06.3 视觉色相对调**：基于"珊瑚红出现概率应该更高，白色为次级"反馈，LED 主读数（FLOOR / `¥` / `#money` / 任务行 / 沉浸态资产）回到 `--hud-readout` (`#f8a59b` 暖珊瑚)，冷白降级为标签 / 配额副数字 / mission 标签等次级辅助。本节配额过线 mint 极端态**完全不变**（`--quota-cleared` / `--quota-flash` / 场景层 `// FBC §3.1 — 配额已达成` 闪现仍是 mint，奖励信号识别度更高 —— 因为周边环境从全冷白变成了暖珊瑚）。详见 `docs/设计规范-Design-System.md § 9 / 附录 A.4` 与 `docs/0.06版本更新产品功能.md § 八`。

### 0.5 债务撤离入账（升级 C-1）

`forceDebtCashOut()` 现写入档案：

```
careerMerge({ creditsCollected: payout, debtCashout: true })
```

- `payout = floor(this.credits)`（玩家被划扣后实际取回额）计入 `total_credits_collected`，使 NET P&L 真实反映"投入 - 取回"亏空，不再被动隐藏。
- `total_debt_cashouts += 1`：与 `total_liquidations` 区分开来，便于审计"未达配额的低损撤离"。
- 不写 `creditsLost / liquidation`：损失自然体现在 `total_buy_in - total_credits_collected` 的差。

### 1.2 字段定义与累积公式

| 字段名（JSON） | 含义（玩家侧） | 写入 / 更新公式 |
|----------------|----------------|----------------|
| `total_credits_collected` | 累计已核销资产：多局「成功撤离」到账金额之和 | `+= floor(payout)`，在 `GameController.cashOut()` 成功路径调用 `careerMerge({ creditsCollected: payout })`。 |
| `total_floors_climbed` | 累计攀爬层数 | `+= floorsJumped`（单次上行随机跳跃的层数之和），在 `processAscendComplete()` 调用 `careerMerge({ floorsClimbed, floorReached })`。 |
| `max_floor_reached` | 历史最高深度（楼层） | `= max(原值, 当前局 this.floor)`，同上 `careerMerge({ floorReached: this.floor })`。 |
| `total_liquidations` | 清算次数 | 每发生一次记入档案的清算 `+= 1`：`finishReveal()` 中未缓和 **LIQUIDATION**；`_autoLiquidate()`（Hiss 超时）。 |
| **`total_committee_overrides`**（v0.05 新增） | FBC 代决议触发次数（封存货舱选盘倒计时归零自动开柜的累计） | `+= 1`，在 `_resolveLockerSelection(committeeOverride=true)` 内 `careerMerge({ committeeOverride: true })`。 |

**加载归并**（`careerLoad`）：解析失败或缺键时按 0 处理；`max_floor_reached` 与历史取 **max**；其余数值键缺失视为 0。旧版 0.04 存档无 `total_committee_overrides`，读时回退 0，向后兼容。

**持久化**（`careerMerge`）：`careerLoad()` → 应用 `delta` → `_careerSave` 写回同一键。

## 3. 选盘 / 货舱 / 代决议（权力制衡的新主回路）

### 3.1 柜数与倒计时

| Biome | `lockerCountByBiome` 默认 | 养猪期（floor ≤ 3）实际 | 倒计时 = `count * 4000`，夹在 [6000, 16000] |
|-------|---------------------------|--------------------------|-----------------------------------------------|
| `admin`（1–10 楼） | **2** | **2** | 8000 → 6000（撞下限被裁到 8000 不变）｜实际 8s |
| `maintenance`（11–25 楼） | **3** | — | 12000 |
| `supernatural`（26+ 楼） | **4** | — | 16000（撞上限） |

> Biome 由 `biome.ranges` 推导；`_resolveLockerCount` 还会将柜数夹到 [2, 4]。

### 3.2 候选生成保证

`generateLockerHand(ctrl, count, quotaShift)` 内三重兜底：

1. 主循环 `count*6` 次重抽，每次独立 `rollEventFromR(r, ...)`；
2. `minBidGapPositive=1`：若 N 次抽完仍无 POSITIVE/DOUBLE，则把其中一只候选替换为正面带的拟造 outcome（`r ∈ [NEGATIVE_MAX, POSITIVE_MAX)`）；
3. 柜号 `numberMin..numberMax` 区间随机不重复，最大重抽次数 `max(40, count*12)`；若仍不足则按顺序补齐——避免确定性 RNG 测试场景下死循环。

### 3.3 FBC 代决议（committee override）

| 字段 | 默认 | 作用 |
|------|------|------|
| `fbcOverride.creditsClampRatio` | **0.62** | 代决议下账面折扣率：净增益 × ratio；净亏损 × `(1 + (1 - ratio))`，亏损反向放大 |
| `fbcOverride.extraCorruption` | **0.45** | 选盘后追加的偏移指数（与 `corruption.onCommitteeOverride` 同源，控制器实际累加 `extraCorruption`） |
| `corruption.onCommitteeOverride` | **0.45** | 与上同步；改一处时强烈建议同步另一处 |
| `fbcOverride.banner` | `FBC · 委员会代决议 · 强制开启 LOT-{{lot}}` | 浮层关闭后的委员会闪条文案 |
| `fbcOverride.messages[]` | 5 条 | 备用回音文本，目前未在 UI 直接消费，留作叙事扩展 |

代决议触发后：

- `outcome.committeeOverride = true`、`outcome.committeeOverrideLocker = lockerNumber`；
- 折扣 / 加偏移在乘客修正之后执行（避免 `creditsMod` 与 ratio 同向叠加破坏意图）；
- 生涯 `total_committee_overrides += 1`；
- 音频侧调 `audio.playCommitteeOverride()`（`AudioEngine` 桩与实现都接住）；UI 侧调 `playFbcOverrideStamp()` SFX 与 `flashCommitteePassengerDirective(banner)`。

### 3.4 选盘 SFX 时机表（v0.05）

| 时机 | 方法 | 触发逻辑 |
|------|------|----------|
| 浮层 `.active` 立即 | `playLockerSelectionEnter` | `showLockerOverlay()` 末尾 |
| 玩家点击柜门 | `playLockerOpen` | `handleLockerClick()` 调用 `selectLocker(idx)` 之前 |
| 倒计时进入 panic | `playLockerPanic` | `startLockerCountdown` 内首次 `ratio ≤ 0.3`（一次性） |
| FBC 代决议盖章 | `playFbcOverrideStamp` | 倒计时归零分支调用 `autoSelectLocker()` 之前 |
| 货舱清点单件揭露 | `playManifestRevealItem(polarity)` | `runManifestReveal` 每行 `revealed` 时根据 `delta>0/=0/<0` 取 `gain/neutral/loss` |

所有 SFX 程序化合成、直连 `master`，**绕过 BGM 总线**，因此即使在 BGM 关闭模式下也清晰可闻。

### 3.5 货舱清点：金额拆分与文案抽取

| `outcome.kind` | `splitPattern` 默认 | 三件极性提示 |
|----------------|----------------------|----------------|
| `POSITIVE` | `[0.25, 0.35, 0.40]` | 通常全 gain；负数模拟时按符号判定 |
| `DOUBLE` | `[0.20, 0.30, 0.50]` | 全 gain，最后一件最大 |
| `NEGATIVE` | `[0.30, 0.30, 0.40]` | 通常全 loss；最后一件吸收四舍五入误差 |
| `LIQUIDATION` | `[0.0, 0.0, 1.0]` | 引擎跳过清点（`runManifestReveal` 早返回 0），仅保留模式以备后续动效 |
| 其它 | `fallbackSplit = [0.33, 0.33, 0.34]` | 三件均分 |

文案池 `manifestPool[biome][polarity]` 共 6 类组合（admin/maintenance/supernatural × positive/negative/neutral），权重无放回抽取（`taken` 表去重，全空时退到完整池）。

## 4. 事件概率与权重

### 4.1 偏移指数（Corruption）

| 来源 | 默认值 | 备注 |
|------|--------|------|
| `corruption.max` | **3.0** | UI 条 = `corruption / 3` |
| `liquidationShift` | **0.05** | 清算带含 `min(corruption × 0.05, 0.20)` |
| `perFloor` | **0.08** | 每跳一层；CORRUPTION_SURGE 时 ×2 |
| `onPositive` / `onNegative` / `onDouble` | **0.05 / 0.12 / 0.35** | 结果带累积 |
| `onVipAuditRelief` | **−0.11** | 审查首位 VIP |
| `onDangerUnveil` | **+0.17** | 审查 DANGER 揭穿 |
| **`onCommitteeOverride`**（v0.05） | **+0.45** | FBC 代决议追加 |

**HUD 文案分档**（仅展示用）：`<20%` 稳定；`<42%` 波动；`<65%` 偏高；否则 **临界**。

### 4.2 乘客身份权重（5 类）

| id | weight | label | 关键修正 / 选盘行为 |
|----|--------|-------|---------------------|
| VIP | 28 | 监理员 | `creditsMod=1.15`、阈值降 |
| SCAMMER | 28 | 红圈商人 | `creditsMod=0.85`、100% 伪装 |
| DANGER | 22 | 希斯潜伏者 | volatile、50% 伪装 |
| **INFORMER**（v0.05） | 14 | 内线情报员 | 阈值/资本中性；`lockerHintChance=0.85` · `lockerHintAccuracy=0.78` |
| **CLEANER**（v0.05） | 8 | 清扫员 | 阈值/资本近中性；`lockerHintChance=0.45` · `lockerHintAccuracy=0.55` |

**情报抽取**（`_computeLockerHint`）：选盘前每位非伪装乘客掷一次 `lockerHintChance`；命中者按 `lockerHintAccuracy` 取最高的一位为情报源。然后：

1. 在候选柜中查找首个 POSITIVE/DOUBLE（goodIdx）与首个 NEGATIVE/LIQUIDATION（badIdx）；
2. 50% 偏向播报"好情报"，否则"坏情报"；
3. `truthful = rng() < lockerHintAccuracy`：真情报指向真实柜，假情报指向相反极性柜（无对手柜时随机）；
4. 文案从 `lockerHintCopy.good` / `bad` 池均匀抽取，`${locker}` 替换为 `01..99`。

### 4.3 选盘候选独立掷骰

每只柜独立调 `rollEventFromR`，使用同一份 `corruption / passengers / activeEnvEvent / quotaShift / floor` 上下文。`outcome.creditsMultiplier` 在生成时已固化，`selectLocker` 时不会重新掷骰，仅施加副作用（乘客修正、volatile、代决议折扣等）。

### 4.4 Lore（开门叙事 / 异动档案）

**来源**：`js/game-controller.js` · `OUTCOME_NARRATIVE_POOL` + `getOutcomeNarrative(outcome)`。
**抽取规则**：先按本层 **主结果类型**（DOUBLE / POSITIVE 高或低倍 / GOLDEN / NEGATIVE / LIQUIDATION）选子池，再在子池内 **均匀随机** 一条（`floor(rng * pool.length)`），**无 weight 数组**。

| 子池 | 条目数 | 条件（摘要） |
|------|--------|--------------|
| DOUBLE | 3 | `outcome.kind === 'DOUBLE'` |
| POSITIVE_HIGH | 3 | POSITIVE 且倍率 ≥ ~1.16 |
| POSITIVE_LOW | 2 | POSITIVE 且未达高倍 |
| GOLDEN_FLOOR | 2 | POSITIVE 且 `goldenFloor` |
| NEGATIVE | 3 | NEGATIVE |
| LIQUIDATION | 1 | LIQUIDATION |

每条含 `tag`、`text`、`lore`；`lore` 供档案模态与三位一体「查阅异动」链路使用。

> 0.05 暂未为 FBC 代决议专门写一组 lore，而是复用上述子池，再叠加 `fbcOverride.banner` 横幅；后续可在 `OUTCOME_NARRATIVE_POOL` 增加 `COMMITTEE_OVERRIDE` 子池供模态消费。
