# 《暴富电梯》0.06 版本经济改造（House Edge 第一阶段）

> **版本**：0.06.3（增量发布，与 0.05 主回路完全兼容）
> **生成日期**：2026-04-28
> **里程碑序**：0.06 经济曲线（① 分段衰减 + ② FBC Edge）→ 0.06.1 系统弹窗字号亮度规范化 → 0.06.2 HUD LED 去 mint + 配额标签清理 → **0.06.3 LED 主 / 次级色相对调（暖珊瑚回到主色 / 冷白降级辅助）**（详见八章）。
> **说明**：本版本只改"经济曲线 + UI 视觉"，不动状态机 / 选盘 / 清点回路。
> 所有改动可通过 `fbcEdge.enabled=false` / `baselineGrowth.tiers=null` / 单独 git revert v0.06.2/v0.06.3 视觉补丁 一键回退。

## 一、问题背景

0.05 阶段反馈集中在两点：

1. **赢钱过于线性**：开局 ¥200 → 几千 / 上万只需 5–6 层。
2. **破产无痛**：清算只让本局归零，玩家"白送的杠杆"还回去而已。

数学上，单层 RTP（玩家回报率）≈ **127%**，House Edge ≈ **−27%**——
这意味着游戏不是赌场，是一台正期望永动机。

0.06 第一阶段（① + ②）旨在把单层 RTP 拉回 **96–98%** 区间，
为后续的"信用评级 / 浮盈折现 / 撤离税"等阶段铺垫。

## 二、改造 ① — `baselineGrowth` 分段衰减

**配置**：`config/game-events.json · baselineGrowth.tiers`
**代码**：`js/game-controller.js · _applyBaselineGrowth` / `pickBaselineGrowthTier`

| stakeRatio = credits / quota | 衰减后倍率 | 标签 |
|---|---|---|
| < 1.5 | 1.10 | `incentive` |
| < 3.0 | 1.06 | `throttle` |
| < 6.0 | 1.03 | `freeze` |
| ≥ 6.0 | 1.015 | `millionaire-trap` |

**效果**：
- 早期未达配额：保持原 1.10，玩家上头期不被破坏。
- 资产到 quota×1.5 后：每层增长被压到 +6%。
- 资产到 quota×6 后：每层只涨 1.5%——**守住比赚到更难**，逼迫玩家做撤离决策。

**回退**：`baselineGrowth.tiers` 留空或字段非法时自动回退到 `baselineGrowth.multiplier`（1.10 常数），与 0.05 行为完全一致。

## 三、改造 ② — FBC 联邦资产流通损耗（House Edge）

**配置**：`config/game-events.json · fbcEdge`
**代码**：`js/game-controller.js · _applyOutcomeToCredits` / `pickFbcEdgeRate`
**UI**：`index.html · showDoorResultFrame` 三位一体帧底部新增一行极细官僚注脚

| 结果带 | 默认税率 | 含义 |
|---|---|---|
| POSITIVE | 4% | 普通正面收益 |
| DOUBLE | 8% | 翻倍：大赢大税 |
| GOLDEN（POSITIVE 且 `goldenFloor`） | 12% | 肥尾另算 |
| NEGATIVE / LIQUIDATION | 0% | 输了不再加税 |

**抽水时机**：在 `speculator` / `blindConfidence` 等正向乘子之后，
语义为"无论你怎么把 delta 抬高，FBC 都先切一片"。

**视觉表达**：

```
[ A-007 三联印章 ] | +¥320
// FBC §4.7 已扣 ¥13 · 4% 流通损耗
```

注脚使用 Courier New / 0.5–0.6rem / 透明度 55%，与 FBC 官僚美学一致；
仅在正向收益且税额 > 0 时显示。

**回退**：`fbcEdge.enabled=false` 时整段 no-op。

## 四、综合 RTP 推算（落地后）

| 阶段 | RTP 计算 | 累计 RTP |
|---|---|---|
| 0.05 当前 | 1.154 × 1.10 | **127%** |
| ① baseline 分段衰减（中等期） | 1.154 × 1.06 | **122%** |
| ② FBC Edge | 122% × 0.96 | **117%** |

> 单仅 ① + ② 阶段就把 RTP 从 127% 压到 117%，House Edge 改善 **10 个百分点**。
> 后续阶段（③ 偏移分段强化 / ④ 信用评级 / ⑤ 浮盈折现 / ⑥ 撤离税）会继续把高资产玩家的 RTP 推到 91–96% 区间。

## 五·补 — 生涯档案 · 博彩对账三件套

### 立项动机

0.05 之前，战绩页只有"累计核销资产"和"破产次数（计数）"。
玩家诉求：**破产对应没有具体金额，无法判断"我到底是赢还是亏"**——
这是博彩流水账的最基本诉求（buy-in / cash-out / net P&L）。

### 字段扩展

`fbc_career_archive_v1` 在原 5 字段之上新增 **2 项**：

| 新字段 | 含义 | 写入点 |
|---|---|---|
| `total_buy_in` | 累计建仓投入（每局 `initialBet` 之和） | `index.html · btnDeploy.click` 后 `GameController.careerMerge({ buyIn: amount })` |
| `total_credits_lost` | 累计破产损失（清算事件中累计该局 `initialBet`） | `js/game-controller.js · finishReveal/_autoLiquidate` 与 `liquidation:true` 同 delta 写入 |

衍生量（UI 实时计算，不入库）：

```
net P&L = total_credits_collected − total_buy_in
```

### 战绩页 UI 改版

右上角 ≡ 菜单·战绩页 5 行布局：

| 行 | 字段 | 视觉 |
|---|---|---|
| 1 | `累计建仓投入` / `累计核销资产` 并列 | 暗金 / FBC 暖色 |
| 2 | `综合损益结算 · NET P&L`（通栏） | 正绿 / 负红 / 平 |
| 3 | `历史最高深度` / `累计上行楼层` 并列 | — |
| 4 | `破产记录: N 次 · 损失 ¥X`（通栏） | `>=3 次` 红色 |
| 5 | `FBC 委员会代决议次数`（通栏） | — |

### 兼容性

- 旧档案（无 `total_buy_in` / `total_credits_lost`）：`careerLoad` 自动回退 0；首次打开战绩页时 PnL 等于"累计已核销资产"全额（视为净盈展示）。
- 下次 deploy 后即恢复对账，老玩家无感升级。

### 一键回退

如需关闭对账系统：
- `careerMerge` 不调用 `buyIn` / `creditsLost` 不影响主链路（其余字段照旧累加）。
- 战绩页可通过删除 `renderCareer` 中第 1 / 2 / 4 行的 HTML 块回到 0.05 视觉。

> 本系统不修改任何概率 / 杠杆 / 奖惩公式，仅扩展数据可视化层，与改造 ① / ② 完全正交。

## 五·二 — 配额系统三件套（升级 A / B / C-1）

### 立项动机

0.06 ① ② 把 RTP 压回赌场区间后，配额本身的反馈变得突出：

1. **静态目标**：所有杠杆都 `quota = ¥1,000`，¥200 玩家要赚 5×、¥1,500 玩家只要 0.67×，难度不一致。
2. **过线无感**：跨过 quota 线只播一个音效，没有视觉、没有数值奖励，"任务"消失感强。
3. **未达 quota 撤离**：`forceDebtCashOut` 真实地划扣过 credits，却不写入战绩，造成 NET P&L 长期失真。

### 升级 A — 动态配额（按 buy-in 分段）

**配置**：`config/game-events.json · quota.tiers`
**代码**：`js/game-controller.js · pickQuotaForBuyIn` / `reset()` 中赋值

| `buyInMax` | `multiplier` | tier label |
|---|---|---|
| 250 | 5.0 | small-stake |
| 600 | 4.0 | standard |
| 1500 | 3.0 | leveraged |
| 99999 | 2.5 | high-roller |

- `quota = round(buyIn × multiplier / 50) × 50`，最小不低于 `quota.minQuota = 500`。
- 暴露 `GameController.pickQuotaForBuyIn` 给授权面板"目标 ¥X · 杠杆 ¥Y"实时预览。
- 回退：`tiers` 留空时自动用 `quota.target = 1000`（与 0.05 一致）。

### 升级 B — 配额过线庆贺

**事件**：`GameController.onQuotaCrossed(payload)`
**触发**：`finishReveal()` 内 `_quotaCrossed` 翻 false→true 的同一帧

UI 反馈集合（`index.html · showQuotaClearedFeedback`，**v0.06.2 已精简**）：

1. HUD `#money` / quota / 分隔符渐染 mint（`--quota-cleared`）+ 1.1s 一次性扫光（`--quota-flash`）。
2. ~~HUD 数字下方一次性 9 px monospace 标签 `// ALLOC_QUOTA · CLEARED`~~ — **v0.06.2 移除**（占用 LED 高度且与第 3 项闪现重复，规则同步删除 `.hud__quota-cleared-tag` 相关 CSS / JS / DOM）。
3. `stage` 顶部 2.6s 闪现 `// FBC §3.1 — 配额已达成 ×<multiplier>`，`backdrop-filter: blur(2px)`。

节拍奖励（`config · quota.crossedBonusCountdownMs = 1000`）：

- 下一次 `LOCKER_SELECTING` 倒计时 +1s（仅触发一次）。
- 落地：`this._quotaBonusPendingMs` → 选盘构造时叠加 → 清零。

战绩档案：`careerMerge({ quotaCrossing: true })` → `total_quota_crossings += 1`。

**回退**：`quota.crossedBonusCountdownMs = 0` 关闭节拍奖励；CSS 类 `--quota-cleared` / `--quota-flash` 不挂即视觉静默。

### 升级 C-1 — 债务撤离入账修复

**触点**：`forceDebtCashOut()`

修复前：玩家走 `forceDebtCashOut` 实际拿走了 `payout = floor(credits)`，但 `careerMerge` 不写入；NET P&L 永远只见 buy-in 出，不见 cash-out 入，"债务撤离"被永久统计为 −100% 损失。

修复后：

```js
careerMerge({ creditsCollected: payout, debtCashout: true });
```

- `total_credits_collected += payout`：与 `cashOut()` 走同一个统计字段，NET P&L 自动反映"投入 - 取回"差值。
- `total_debt_cashouts += 1`：与 `total_liquidations` 区分，便于审计"未达配额的低损撤离"。
- 不写 `creditsLost / liquidation`：损失体现在 `total_buy_in - total_credits_collected` 自然差。

> 所有升级与改造 ① ② 完全正交：未触发过线时主链路与 0.06 ①② 一致；触发后只多一段 UI 反馈与一次档案累加。

## 六、监控建议

发布后埋点观察：

| 指标 | 当前预期 | 阈值 |
|---|---|---|
| 平均单局存活层数 | 8–14 层 | 不低于 8 |
