# 《暴富电梯》全局数值类信息汇总

> **用途**：跨模块数值底表，供调平衡、对文案、写自动化测试断言时快速查阅。  
> **权威顺序**：`config/game-events.json` → `js/game-controller.js` / `index.html` 内 `GameController.cfg` 读取与硬编码兜底。  
> **更新**：与仓库当前逻辑一致；若改字段名或公式，请同步改本节。

## 1. 生涯维度（Director's Dossier）

### 1.1 存储

| 项目 | 值 |
|------|-----|
| **LocalStorage 键** | `fbc_career_archive_v1` |
| **序列化格式** | JSON 对象，UTF-8 字符串写入 |

### 1.2 字段定义与累积公式

| 字段名（JSON） | 含义（玩家侧） | 写入 / 更新公式 |
|----------------|----------------|----------------|
| `total_credits_collected` | 累计已核销资产：多局「成功撤离」到账金额之和 | `+= floor(payout)`，在 `GameController.cashOut()` 成功路径调用 `careerMerge({ creditsCollected: payout })`。 |
| `total_floors_climbed` | 累计攀爬层数 | `+= floorsJumped`（单次上行随机跳跃的层数之和），在 `processAscendComplete()` 调用 `careerMerge({ floorsClimbed, floorReached })`。 |
| `max_floor_reached` | 历史最高深度（楼层） | `= max(原值, 当前局 this.floor)`，同上 `careerMerge({ floorReached: this.floor })`。 |
| `total_liquidations` | 清算次数 | 每发生一次记入档案的清算 `+= 1`：`finishReveal()` 中未缓和 **LIQUIDATION**；`_autoLiquidate()`（Hiss 超时）。 |

**加载归并**（`careerLoad`）：解析失败或缺键时按 0 处理；`max_floor_reached` 与历史取 **max**；其余数值键缺失视为 0。

**持久化**（`careerMerge`）：`careerLoad()` → 应用 `delta` → `_careerSave` 写回同一键。

### 1.3 授权面板「动态评价」（仅展示）

与档案字段联动，**不改变局内概率**（`index.html` · `updateCareerStats`）：

| 条件 | 第三行评价文案 |
|------|----------------|
| `total_liquidations >= 3` | `// 委员会对您的稳定性表示担忧。` |
| 否则若 `total_credits_collected >= 5000` | `// 您正在重塑局内的财务结构。` |
| 否则 | 不展示第三行 |

清算句优先于高资产句。

## 3. 事件概率与权重（权力制衡 + Lore）

### 3.1 卡牌：全局 Reward / Bust 门槛（代码真值）

对每张发到的牌，先按身份取偏置，再掷一次决定进奖池还是罚池：

\[
P(\text{Reward}) = W_{\text{reward}} + b
\]

其中 \(W_{\text{reward}} =\) `cardSelection.rewardTotalWeight`（默认 **0.55**），\(b =\) 该牌 `identity.luckBias`（可负）。  
\(P(\text{Bust}) = 1 - P(\text{Reward})\)（在奖、罚池均非空时；缺池时有回退逻辑见 `generateCardHand`）。

`rewardTotalWeight + bustTotalWeight` 在配置中应 **= 1.0**（`_note`）。

### 3.2 Reward 池内加权（`rewardEvents[].weight`）

| id | weight（默认） | 相对占比（约） |
|----|------------------|----------------|
| windfall | 25 | 23.8% |
| energy_crystal | 30 | 28.6% |
| audit_pass | 20 | 19.0% |
| ghost_tip | 15 | 14.3% |
| time_rift | 8 | 7.6% |
| blessed | 7 | 6.7% |
| **合计** | **105** | **100%** |

在已进入 Reward 分支后，具体事件按上表 **weight 加权** 抽取。

### 3.3 Bust 池内加权（`bustEvents[].weight`）

| id | weight（默认） | 相对占比（约） |
|----|------------------|----------------|
| ghost_tax | 25 | 23.8% |
| curse | 20 | 19.0% |
| trap_floor | 22 | 21.0% |
| hiss_whisper | 18 | 17.1% |
| reality_collapse | 10 | 9.5% |
| soul_drain | 10 | 9.5% |
| **合计** | **105** | **100%** |

### 3.4 身份壳：`identities` 出现率与 luckBias

发牌时从池中 **无放回** 抽 `cardsPerRound`（默认 **3**）个不同 `id`，抽取概率与 `identities[].weight` 成正比。

| id | weight（默认） | luckBias \(b\) | 单卡 \(P(\text{Reward})=0.55+b\)（截断前） |
|----|----------------|----------------|---------------------------------------------|
| elder | 25 | +0.05 | 0.60 |
| child | 20 | −0.08 | 0.47 |
| handsome | 20 | 0 | 0.55 |
| woman | 15 | +0.03 | 0.58 |
| shadow | 10 | −0.12 | 0.43 |
| doctor | 10 | +0.06 | 0.61 |

若 \(P(\text{Reward})\) 超出 \([0,1]\)，仍以 `rng() < P` 实现，极端需靠配置自检。

### 3.5 干预者 Board / Ops / Research（叙事 ↔ 调优映射）

**实现说明**：引擎 **不按** 左/中/右座位分配不同事件池；三张牌各自独立经历「抽身份 → 算 \(P(\text{Reward})\) → 抽池」。下表为 **产品叙事**（与 `产品设计-面向用户.md` 一致）与 **调优时可动的旋钮** 对照，便于「把某方写得更激进/保守」时知道改哪里。

| 代号 | 叙事名 | 建议位次 | 风险偏好（体验） | 建议旋钮（非强制绑定） |
|------|--------|----------|------------------|-------------------------|
| **Board** | 委员会 | 左 | 偏保守、要程序与审计感 | 提高 `audit_pass`、`blessed` 的 **weight**；或提高 **高 luckBias** 身份（如 `doctor`、`elder`）的 **weight**。 |
| **Ops** | 财务部 | 中 | 偏进取、账面波动大 | 提高 `windfall`、`energy_crystal`、`time_rift` 的 **weight**；适度提高 `corruptionDelta` 大的奖励事件以换「甜中带毒」。 |
| **Research** | 调查组 | 右 | 偏极端、情报/异常 | 提高 `ghost_tip`（奖池）与 `hiss_whisper`、`reality_collapse`、`soul_drain`（罚池）**weight**；或提高 **负 luckBias** 身份（`child`、`shadow`）权重以拉高罚池命中率。 |

若未来要在代码层实现「左牌固定 Board 池」，需在 `generateCardHand` 增加按索引分支；当前底表以 **配置 + luckBias** 为唯一杠杆。

### 3.6 Lore（开门叙事 / 异动档案）

**来源**：`js/game-controller.js` · `OUTCOME_NARRATIVE_POOL` + `getOutcomeNarrative(outcome)`。  
**抽取规则**：先按本层 **主结果类型**（DOUBLE / POSITIVE 高或低倍 / GOLDEN / NEGATIVE / LIQUIDATION）选子池，再在子池内 **均匀随机** 一条（`floor(rng * pool.length)`），**无 weight 数组**。

| 子池 | 条目数 | 条件（摘要） |
|------|--------|--------------|
| DOUBLE | 3 | `outcome.kind === 'DOUBLE'` |
| POSITIVE_HIGH | 3 | POSITIVE 且倍率 ≥ ~1.16 |
| POSITIVE_LOW | 2 | POSITIVE 且未达高倍 |
| GOLDEN_FLOOR | 2 | POSITIVE 且 `goldenFloor` |
| NEGATIVE | 3 | NEGATIVE |
| LIQUIDATION | 1 | LIQUIDATION |

每条含 `tag`、`text`、`lore`；`lore` 供档案模态与三位一体「查阅异动」链路使用。
